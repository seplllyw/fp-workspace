# 产品方法论

> 飞派产品团队的标准化产品工作方法。所有 PRD 和产品文档遵循此规范。

---

## 一、产品文档标准结构

所有产品文档必须覆盖以下 11 个章节：

```
1. 背景        — 为什么要做？行业背景、用户痛点、业务驱动
2. 目标        — 做成什么样算成功？量化指标（OKR）
3. 功能        — 具体做什么？功能列表 + 优先级（P0/P1/P2）
4. 业务规则    — 功能背后的业务逻辑和约束条件
5. 交互规则    — 用户操作流程、页面跳转、状态流转
6. 异常        — 错误场景、边界条件、异常处理
7. 权限        — 角色权限矩阵（RBAC）
8. 边界        — 本期不做的事情（明确 scope）
9. 数据        — 数据模型、接口、数据流
10. 流程       — 核心业务流程图/时序图
11. 验收       — 验收标准（AC）
```

## 二、PRD 撰写原则

### 2.1 金字塔原则

```
结论先行
  ├── 核心功能（P0）
  │     ├── 子功能1
  │     └── 子功能2
  ├── 增强功能（P1）
  └── 扩展功能（P2）
```

### 2.2 MECE 原则

- **Mutually Exclusive**：每个功能独立，不重叠
- **Collectively Exhaustive**：所有功能加起来，覆盖完整需求

### 2.3 一个 PRD 只讲一件事

- ❌ 错误："数据资产平台 PRD"（太宽泛）
- ✅ 正确："数据资产平台 · 桩号管理模块 PRD"

## 三、需求优先级定义

| 优先级 | 定义 | 判断标准 |
|--------|------|---------|
| P0 | 必须做 | 不做系统不可用 / 核心链路断裂 |
| P1 | 应该做 | 不做影响体验 / 但有 workaround |
| P2 | 可以做 | 锦上添花 / 无替代方案也不致命 |
| P3 | 暂不做 | 明确列入 backlog，本期不排 |

## 四、Roadmap 规划方法

### 4.1 版本节奏

- **大版本（V1.0 → V2.0）**：季度级，重大功能上线
- **小版本（V1.1 → V1.2）**：月度级，功能增量
- **热修复（V1.0.1）**：周级别，Bugfix

### 4.2 版本号规范

```
V<主版本>.<次版本>.<修订版本>
例：V2.3.1
  ↑   ↑   ↑
  │   │   └── bugfix
  │   └────── 功能增量
  └────────── 重大更新
```

## 五、产品分析框架

### 5.1 页面分析

分析一个页面/模块时，按以下维度：

1. **用户是谁** — 谁在用这个页面
2. **核心场景** — 用户在什么情况下打开这个页面
3. **核心任务** — 用户来这里要完成什么
4. **信息架构** — 页面结构、导航层级
5. **交互流程** — 操作步骤、状态变化
6. **数据流向** — 数据从哪来、到哪去
7. **问题与机会** — 当前痛点、改进空间

### 5.2 竞品分析

1. 竞品选择（直接竞品 / 间接竞品 / 替代品）
2. 对比维度（功能、体验、价格、市场）
3. SWOT 分析
4. 启示与行动项

## 六、周报规范

周报核心原则：**结果导向，数据说话。**

```
本周完成
  ├── 功能A — 已上线（附数据）
  ├── 功能B — 已提测
  └── PRD C — 已完成初稿

下周计划
  ├── 启动功能D PRD
  └── 评审功能E 技术方案

风险与阻塞
  └── 风险1 — 影响范围 — 应对方案

数据看板（可选）
  └── 核心指标本周变化
```
