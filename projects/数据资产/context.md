## 对话总结：桩号管理架构设计 · 头脑风暴

------

### 一、核心需求与最终目标

**最终目标**：产出一份**桩号管理架构设计文档**，面向非技术读者，以"链路、逻辑、流程"为主线，说明桩号管理在智巡 SaaS 体系中的定位、数据流转路径和组织归属关系。

**核心需求**：

1. 桩号管理需要支撑**多行业**——不同行业（高速公路、铁路、水利等）有完全不同的桩号编码体系（如高速用 `K1771+691`，铁路用 `CGJXZQ-3`）
2. 行业属性**不来自数据资产平台自身**，而是来自 **Auth 平台（统一身份认证）** 的组织行业属性
3. 桩号上传后，数据归属到**当前账号所属组织 + 上层组织**
4. 资产管理原型系统是智巡 SaaS "数据管理"模块下的一个子模块，能力从底层数据资产平台获取
5. 原型里可以加入**轻量级的概念示意**（如"当前行业：高速公路"标识、组织层级展示），让看 demo 的人能直观感受到平台的分层逻辑，但不是结构性大改
6. 最终产出是设计文档，不要太技术化，重在讲清楚链路和流程

### 二、已确认的逻辑与进度

#### 2.1 平台分层架构（已对齐）



```
智巡 SaaS 业务应用层
  ├─ 注册/登录
  ├─ 组织管理（创建组织时选择行业 → 行业属性来自 Auth 平台）
  ├─ 态势/工作台
  └─ 数据管理
       ├─ 资产管理 ← 【当前原型在这里】
       ├─ 模型库
       ├─ 地图管理
       ├─ 航线管理
       └─ 媒体管理
           ↓
        API 网关层
           ↓
底层能力平台
  ├─ AI 能力
  ├─ 飞控能力
  ├─ 空间能力
  └─ 数据资产能力（资产模版、桩号基准、编码规则、病害库）
           ↑
        Auth 平台（统一身份认证、组织属性、行业属性）
```

**关键确认**：

- 原型只是智巡 SaaS 中"数据管理 → 资产管理"这一小块
- 行业属性来自 Auth 平台，决定了下层桩号体系的格式和规则
- 中间有 API 网关层做路由和鉴权
- 数据资产能力平台是被调用的能力提供方

#### 2.2 文档组织方式（已确认方案 A）

采用**自顶向下**的结构展开：

1. 平台分层全景（已完成初版校验）
2. 行业层——不同行业对应不同的桩号体系
3. 组织层——组织创建时继承行业基准，桩号数据归属组织及上层组织
4. 桩号数据层——路段→公里桩→百米桩的层级模型
5. 消费层——资产实例如何绑定桩号、巡检/病害如何引用桩号
6. 原型示意方案——轻量标识
7. 当前架构的问题与改进方向

#### 2.3 原型改动范围（已确认方案 B）

在原型中加入**轻量概念示意**，不做结构性重构：

- 桩号管理页顶部加"当前行业：高速公路"标识
- 展示组织层级关系
- 不改 STAKE_DATA 的结构，不改数据流

#### 3.1 当前原型中的桩号数据结构

**全局变量**：



```js
// 租户信息（两级静态 mock）
STAKE_TENANT = {
  currentUserTenant: '昆石高速公路有限公司',
  parentTenant: '云南交投集团'
}

// 桩号字典
STAKE_DATA = {
  tenantName: '昆石高速',
  sections: [{
    id, name, code, startStake, endStake, lengthKm, direction,
    stakes: [{
      id, code ('K1771+691'), shortName, desc,
      stakeType ('百米桩'|'公里桩'),
      lon, lat,
      isBreakStake, prevStake, nextStake,
      routeName ('汕昆高速'), routeCode ('G78'),
      belongTenant, parentTenant, remark
    }]
  }]
}
```

**问题**：

- `STAKE_DATA` 行业硬编码为高速公路（G78 昆石）
- `STAKE_TENANT` 是静态 mock，仅两级
- 桩号数据与资产实例的桩号范围是两个独立子系统，没有联动校验
- 无后端，全部在浏览器内存
