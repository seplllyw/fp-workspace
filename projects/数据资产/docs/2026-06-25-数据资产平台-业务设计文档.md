## 第一部分：平台全景

### 1.1 数据资产平台在智巡 SaaS 中的位置

数据资产平台不是独立系统，而是智巡 SaaS 体系中的**资产台账能力提供方**。它在两个层面存在：

**在 SaaS 应用层**：「数据管理 → 资产管理」菜单项，这是用户看到和使用的前台界面。

**在底层能力层**：作为数据资产能力平台，被 SaaS 层和其他业务产品调用，提供资产模版、桩号基准、编码规则、病害库等核心能力。

```
智巡 SaaS 业务应用层
  ├─ 注册/登录（选择行业 → 初始化配置）
  ├─ 组织管理（行业属性来自 Auth 平台）
  ├─ 工作台 / 整体态势
  └─ 数据管理
       ├─ 资产管理 ← 【数据资产平台前台】
       ├─ 模型库
       ├─ 地图管理
       ├─ 航线管理
       └─ 媒体管理
           ↓
        API 网关层（路由、鉴权、限流）
           ↓
底层能力平台
  ├─ 数据资产能力 ← 【数据资产平台后台】
  ├─ AI 能力（实时识别、视频分析）
  ├─ 飞控能力（DJI 对接、航线执行）
  └─ 空间能力（地图、坐标转换）
           ↑
        Auth 平台（统一身份认证、组织属性、行业属性）
```

**开发要记住的核心关系：**
- 资产管理是智巡 SaaS 的**一个功能模块**，不是全部
- 行业属性**不来自数据资产平台自身**，来自 Auth 平台的 `system_industry`
- 数据资产平台依赖 Auth（拿行业/组织）、飞控（拿任务状态）、AI（消费病害识别结果），**被**运营产品（卫瞳/智巡）消费

### 1.2 多行业体系：一切配置的起点

**核心链路：**

```
用户注册 → 选择行业（如"高速公路"）
  → Auth 下发 industry_id 和行业属性
  → 智巡 SaaS 按行业加载预设的业务场景（路面巡检、交安设施巡检…）
  → 数据资产平台按行业加载：
      • 资产模版（高速有桥梁/边坡/隧道，铁路有轨道/接触网/信号）
      • 桩号编码体系（K1771+691 vs CGJXZQ-3）
      • 病害类型库（高速病害 vs 铁路病害）
      • 编码规则引擎（不同行业编码段不同）
```

**为什么行业属性这么重要：**

行业不是简单的一个"筛选条件"。切换行业意味着：
- 完全不同的资产类型模版（你不需要在高速行业里看到"接触网"）
- 完全不同的桩号格式和校验规则
- 完全不同的病害类型和检查标准
- 不同的编码规则

**开发实现上要注意：**
- `industry_id` 来自 Auth 平台的 `system_industry` 表，DA 不自己定义行业
- 租户可以绑定多个行业（`industry_ids`），但用户在一个组织上下文中一次只操作一个行业
- 行业种子数据（模版/病害/编码规则）在租户首次选择行业时自动初始化灌入
- `industry_id` 存在于资产主表和检测算法配置表中，**不作为**网关 Header 或数据权限维度

### 1.3 五大子系统及其关系

数据资产平台内部由五个核心子系统组成：

| 子功能 | 做什么 | 依赖谁 | 被谁依赖 |
|--------|--------|--------|----------|
| **资产模版** | 定义资产的数据结构、树形构件分解、台账字段 | 编码引擎（绑定编码规则）、病害库（检查位置关联病害） | 在管资产（创建实例时继承模版） |
| **在管资产** | 基于模版创建实例，管理数字档案、巡检记录 | 资产模版（继承结构）、桩号管理（绑定位置） | AI（巡检目标）、运营产品（病害挂接目标） |
| **桩号管理** | 路段→公里桩→百米桩的维护，多行业桩号体系 | Auth（行业→桩号格式）、组织（数据归属） | 在管资产（位置绑定）、巡检（任务范围） |
| **病害库** | 病害类型定义（名称/编码/等级），与检查位置关联 | —（相对独立） | 资产模版（检查位置引用病害）、AI（识别结果映射） |
| **编码引擎** | 按资产类型生成实体编码和构件编码 | 资产模版（编码规则配置）、桩号（桥梁编码含路段码） | 资产模版（绑定规则）、在管资产（自动生成编码） |

**依赖关系图：**

```
Auth 平台 ──→ 桩号管理 ──→ 在管资产 ←── 资产模版 ←── 编码引擎
                 │                              │
                 └──────── 媒体库/病害库 ←──────────────┘
                              ↑
                          AI 平台（病害识别结果映射到病害库标准类型）
```

## 第二部分：核心业务模块

### 2.1 资产模版

**模版是什么：** 资产的"类定义"。就像面向对象编程里的 Class，定义了这类资产长什么样、由哪些构件组成、要记录哪些信息。

**三要素：**

1. **台账字段**：可自定义的扩展字段。例如桥梁模版有"桥长""跨径""设计荷载"，边坡模版有"坡高""岩土类型""防护形式"。字段类型支持文本、数字、单选、多选。

2. **树形结构**：层级化的构件分解，三种节点类型：
   - **分组**：纯组织层级，用于归类。例如"上部结构"下面挂"主梁""横隔板"。分组节点**不可挂数据**。
   - **填写项**：叶子节点，可维护具体数据。例如"桥墩编号""设计桩号"。
   - **检查位置**：叶子节点，需要关联病害库。例如"墩身裂缝检查"需要关联病害类型"裂缝（1级/2级/3级）"。

3. **编码规则**：每个模版绑定实体编码规则（资产实例编码怎么生成）和构件编码规则（构件怎么编号）。

**关键业务规则：**
- 模版创建后，树形结构**已经有资产实例在使用时不能随意大改**——否则已有实例的数据结构会乱。系统需要提示"模版已更新"并引导用户手动迁移。
- 检查位置节点**必须至少关联一个病害类型**。
- 树节点的路径编码（nodeCode）由编码引擎**自动生成**，不需要用户手填。
- 同一资产类型（如"桥梁"）可以有多套模版（如"拱桥模版""梁桥模版"），取决于业务需要的细分程度。

### 2.2 在管资产

**资产实例是什么：** 模版的实例化 + 真实业务数据 + 巡检记录。

**三块内容：**

1. **台账信息（基本信息 Tab）**：继承模版的台账字段定义，填写这座具体桥梁/这段具体边坡的实际值。

2. **数字档案（结构 Tab）**：继承模版的树形结构，实例化为真实构件。例如模版里定义了"桥墩"这个检查位置，在某个具体桥梁的实例里，用户创建出"左幅 1#墩""左幅 2#墩""右幅 1#墩"——这些就是**最小维护单元**。每个单元可以记录：
   - 位置描述、当前状态、备注
   - 关联的病害记录
   - 飞行巡检档案（哪个架次巡检了这个单元）

3. **桩号绑定**：每个资产有起终点桩号范围，用于地图定位和巡检任务分配。

**关键业务规则：**
- 创建资产**必须选择模版 + 填写桩号范围**。
- 最小维护单元是**巡检的最小粒度**：一个单元 = 无人机一次拍摄的目标。例如桥梁巡检不是巡检"整座桥"，而是巡检"1#墩的墩身裂缝"。
- 模版改了结构后，已有实例的处理方式：提醒用户"模版已更新"，由用户决定是否迁移。不自动强制更新。
- **桩号范围的作用**：当巡检任务说"巡检 K1771+000 到 K1775+000"，系统通过桩号范围筛选出这个区间内所有资产。

### 2.3 桩号管理

**桩号是什么：** 高速公路（及其他线性工程）的里程标记系统。K1771+691 表示从起点算起 1771 公里又 691 米的位置。

**三级层级：**

```
路段（如：昆石高速 G78 K1771+000 ~ K1790+000）
  ├─ 公里桩 K1771
  │   ├─ 百米桩 K1771+100
  │   ├─ 百米桩 K1771+200
  │   └─ ...
  ├─ 公里桩 K1772
  │   └─ ...
  └─ ...
```

**多行业桩号格式：**

| 行业 | 桩号格式 | 示例 |
|------|----------|------|
| 高速公路 | K + 公里数 + 米数 | K1771+691 |
| 铁路 | 工区编号 + 分段 | CGJXZQ-300 |
| 水利 | 渠道桩号 / 管线桩号 | 各行业自定 |

**关键业务规则：**
- 行业属性决定桩号格式——**来自 Auth 平台**，DA 不自己定义行业。
- 桩号数据归属到组织：当前组织创建的数据，**父组织可见**。
- **断链桩号**：路线改线后桩号可能不连续（K1772 之后直接跳到 K1775）。需要标记 `isBreakStake=true`。
- 桩号导入（Excel）后需要校验：经纬度范围是否合理、桩号是否连续、是否与已有路段重叠。
- DA 需要桩号维护能力（CRUD、Excel 导入导出、树形展示），但**底层坐标主数据复用 `stake_number_info`**，不重复建设坐标表。

### 2.4 病害库

**两层概念，不要混淆：**

| 概念 | 是什么 | 存在哪 | 举例 |
|------|--------|--------|------|
| **病害类型** | 病害的分类定义（名称/编码/等级） | DA 的 `DEFECT_POOL` / `dcs_asset_exception_type` | "裂缝"（1级/2级/3级）"锈蚀""位移" |
| **病害实例** | 某次巡检中实际发现的病害 | 运营产品 `dcs_biz_exception_event` | "2026-06-25 飞行发现 K1771+691 处 1#墩墩身 2级裂缝" |

**DA 管病害类型，运营产品管病害实例。**

**关键业务规则：**

- 病害类型库按行业预置（高速公路有一套标准病害，铁路有另一套）。
- 检查位置节点（在模版树里）只能关联病害库里已定义的病害类型。
- AI 识别出来的病害结果，需要映射到病害库里的标准类型（如 AI 识别"疑似裂缝"→ 映射到病害类型"裂缝-2级"）。
- 病害实例（AI 识别出来的具体裂缝）以运营产品库为权威，DA **不建病害实例表**，通过挂接表（`da_ops_event_asset_link`）建立"哪个异常属于哪个资产的哪个构件"的关联。

### 2.5 编码引擎

**两层编码：**

**实体编码（资产实例唯一 ID）：**

| 资产类型 | 编码格式 | 示例 |
|----------|----------|------|
| 桥梁 | H + 省码 + 路段码(4位) + L + 流水号(4位) | H78530126L9090 |
| 交安设施 | G78-里程桩号-设施大类码-流水号 | G78-K1852+000-JTBZ-001 |
| 隧道 | TUN-G78-简称-起终点桩号-U/D | TUN-G78-XLK-K1850-K1855-U |
| 边坡 | 管养单位标准编码 | — |
| 服务设施 | SFZ-G78-中心桩号-类型缩写 | SFZ-G78-K1852-FWQ |
| 避险车道 | KD + 流水号 | KD0001 |

**构件编码（资产内部构件编号）：**

```
LS-QD-L-0001
│  │  │  │
│  │  │  └─ 流水号（第1号）
│  │  └─ 方向（左幅）
│  └─ 部位（桥墩）
└─ 桥型（梁式桥）
```

**关键业务规则：**
- 编码 = 多个**编码段**拼接，每段可配置来源：流水号（自增）、外部引用（桩号、路段码、组织码）、固定值（资产类型缩写）。
- 编码在资产保存时**自动生成**，一旦生成**不可修改**——保证后续引用稳定性。
- 不同资产类型有**完全不同的编码规则**，由模版绑定的编码规则配置决定。
- 编码需要保证**全局唯一性**（跨组织、跨行业）。

## 第三部分：关键业务规则

### 3.1 多行业初始化配置流程

这是平台支持多行业的**核心链路**，开发特别容易简化理解成"加个 industry_id 字段就行"。

**完整流程：**

1. **租户注册**：用户在 Auth 平台注册租户，选择行业（可多选，如同时选了"高速公路"和"铁路"）。
2. **组织创建**：租户下创建组织，组织继承租户的行业范围。创建一个"高速公路管理处"组织，它的行业属性就是"高速公路"。
3. **首次进入**：组织下的用户首次进入智巡 SaaS → 数据管理 → 资产管理时，系统检测该组织是否已初始化：
   - 如果未初始化 → 自动灌入该行业的**种子数据**：
     - 资产模版（高速：桥梁模版/边坡模版/交安模版/隧道模版…）
     - 病害类型库（高速标准病害）
     - 编码规则配置
     - 默认路段信息
   - 如果已初始化 → 直接进入资产管理界面
4. **日常使用**：用户在当前行业上下文中创建模版、录入资产、维护桩号。所有操作都带有 `industry_id` 标记。

**切换行业：** 如果一个租户有多个行业（比如同时做高速和铁路），用户需要切换到另一个行业的组织才能看到该行业的资产体系。不是简单切换下拉框，而是**完全不同的数据视图**。

### 3.2 桩号与资产的关联

**当前状态（原型）：** 桩号数据和资产数据是两个独立子系统，各自维护，**没有联动校验**。

**未来需要实现：**

1. **创建资产时校验**：用户填写的桩号范围（如 K1771+000 ~ K1771+600）必须在已有路段范围内。
2. **桩号变更时提示**：如果某段桩号被修改或删除，系统提示"以下资产引用了该桩号范围"，防止数据断裂。
3. **巡检任务关联**：巡检任务绑定桩号区间后，系统自动筛选出该区间内的所有资产作为巡检目标。

**桩号的双重身份：**
- 作为**空间基准**：桩号对应经纬度坐标（来自 `stake_number_info`），提供 GIS 地图定位。
- 作为**业务主键**：资产台账、巡检记录、病害报告、养护工单都引用桩号来标识位置。

### 3.3 病害库与 AI 识别的关系

这是 DA 和 AI 平台之间最重要的集成点：

```
AI 识别出"疑似裂缝" 
  → 映射到病害类型库中的"裂缝" 
  → 根据严重程度判定等级（1级/2级/3级）
  → 病害实例写入运营产品库 dcs_biz_exception_event
  → DA 通过挂接表 da_ops_event_asset_link 关联到具体资产和构件
  → 用户在运营产品中审核确认
  → 已审核样本回流到 AI 平台做训练
```

**DA 在这个链路中的角色：**
- 提供**病害类型库**（AI 识别结果的标准参照）
- 提供**构件编码对照表**（AI 知道识别到的裂缝属于"梁式桥-桥墩-左幅"）
- 提供**挂接关系**（这个异常属于哪座桥的哪个构件）
- **不建病害实例表**，不重复存储 AI 识别结果

### 3.4 九阶业务闭环（DA 视角）

整个智巡业务是一个闭环，DA 参与其中的①④⑥⑨：

```
①数据资产 → ②巡检 → ③AI检测 → ④病害 → ⑤审核 → ⑥整改 → ⑦标注 → ⑧训练 → ⑨精度提升
    ↑DA权威    ↑DA计划   ↑AI      ↑DA挂接   ↑运营    ↑DA台账   ↑AI     ↑AI      ↑DA报表
```

| 阶段 | DA 做什么 | DA 不做什么 |
|------|-----------|-------------|
| ① 数据资产 | 资产/构件建档、RTK 映射、行业算法配置（**权威**） | — |
| ② 巡检 | 巡检计划编制、归档（执行在运营产品） | 不执行飞行任务 |
| ③ AI 检测 | 消费资产/构件/行业配置 | 不做推理 |
| ④ 病害 | 异常→资产→构件的**挂接** | **不建病害实例表** |
| ⑤ 审核 | 只读/Feign 查询审核状态 | 不做审核操作 |
| ⑥ 整改 | 养护工程记录（**权威**） | 养护工单执行在运营产品 |
| ⑦⑧⑨ | 汇总报表（挂接率/召回率/精度趋势） | 不做标注、训练 |

### 3.5 与外部平台的集成边界

| 外部平台 | DA 需要什么 | DA 提供什么 | 集成方式 |
|----------|-------------|-------------|----------|
| **Auth 平台** | 行业属性、组织树、用户权限、租户信息 | —（纯消费） | Feign / 共享库只读 |
| **飞控平台** | 飞行任务状态、航线信息 | 资产位置（桩号→坐标），供航线规划 | API 调用 |
| **AI 平台** | 推理结果（病害识别）、算法配置 | 病害类型库、构件编码对照表、标注样本 | ETL + MQ |
| **运营产品**（卫瞳/智巡） | 巡检任务、异常事件、巡航媒体 | 资产台账、挂接关系（异常→资产→构件） | ETL ODS + API |

**核心原则：**
- DA **不重复建设**运营产品的航线/任务/异常/媒体能力
- DA 是资产台账的**权威源**，其他系统引用资产的统一 ID
- 病害实例以运营产品库为权威，DA 只做**挂接**（哪个异常属于哪个资产的哪个构件）
- 多产品通过 `source_product_code`（`feipai-wt` / `feipai-zx`）区分来源

## 第四部分：原型参考说明

### 4.1 原型定位

`prototype/index.html` 是一个**纯前端演示原型**，用于验证交互逻辑和业务规则。它不是可交付的代码，但开发可以从中理解：

- 页面结构和导航关系
- 数据结构和字段定义
- 用户操作流程
- 业务规则的前端表达

### 4.2 原型与正式产品的差异

| 维度 | 原型 | 正式产品 |
|------|------|----------|
| 数据存储 | 浏览器内存，刷新即丢失 | PostgreSQL + MySQL |
| 行业支持 | 仅高速公路（硬编码） | 多行业，由 Auth 下发 |
| 桩号数据 | 全局变量 `STAKE_DATA` | 复用 `stake_number_info` + DA 扩展 |
| 组织归属 | 静态 mock 两级 | Auth 平台真实组织树 |
| 多租户 | 不支持 | 租户行级隔离 |
| 资产类型 | 7 大类 | 6 大类 + 预留扩展 |

### 4.3 原型中的关键数据结构

原型中定义了以下全局变量，开发可以参考其字段设计：

- `templates[]` — 资产模版数组
- `instances[]` — 资产实例数组
- `DEFECT_POOL[]` — 病害类型库
- `STAKE_DATA` — 桩号/路段数据
- `STAKE_TENANT` — 租户信息（mock）

这些结构的具体字段定义见 `CLAUDE.md` 中的详细说明。

## 附录 A：术语表

| 术语 | 说明 |
|------|------|
| 模版 (Template) | 资产的"类定义"，定义数据结构、树形构件分解、台账字段 |
| 资产实例 (Instance) | 模版的实例化，代表真实的高速公路资产对象 |
| 台账 (Ledger) | 资产的基本信息记录 |
| 数字档案 (Archive) | 资产的构件分解结构 |
| 最小维护单元 (Unit) | 巡检的最小粒度，如"左幅 1#墩" |
| 树形结构 | 层级化的构件分解树，三种节点类型：分组/填写项/检查位置 |
| 桩号 (Stake) | 公路里程标记，格式如 K1771+691 |
| 百米桩 | 每 100 米一个的桩号标记 |
| 公里桩 | 每 1 公里一个的桩号标记 |
| 断链桩号 (Break Stake) | 路线改线后桩号不连续的情况 |
| 病害 (Defect) | 构件损伤/缺陷，如裂缝、锈蚀 |
| 病害类型 | 病害的分类定义（名称/编码/等级） |
| 病害实例 | 某次巡检中实际发现的病害记录 |
| 航点 (Waypoint) | 无人机巡检航线中的拍摄点位 |
| 架次 (Flight/Mission) | 一次巡检飞行任务 |
| 实体编码 | 资产实例的唯一 ID |
| 构件编码 | 资产内部构件的编号 |
| G78 | 汕昆高速编号 |
| 昆石 | 昆明—石林段 |
| 卫瞳 (WeiTong) | 飞派卫瞳，施工养护巡检产品 |
| 智巡 (ZhiXun) | 飞派智巡，高速公路无人机巡检产品 |
| 飞控 (FeiKong) | 飞行控制平台 |
| Auth 平台 | 统一身份认证、组织管理、行业管理 |
