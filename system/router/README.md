# Router · 任务路由系统

> 默认使用当前模型。当任务明显更适合其他模型时，**提示用户切换**。

---

## 实际运作方式

```
用户任务
  ↓
Claude（当前模型）接收
  ↓
判断：这个任务我擅长吗？
  ├── 擅长 → 直接做 ✅
  ├── 能做但不最优 → 先做，做完提一句"下次可以用 XX 模型"
  └── 明显不擅长 → 提示："这个任务更适合 DeepSeek/MiniMax，要切换吗？"
```

**核心原则：不频繁切换、不打断流程、不强制。**

---

## 切换触发阈值

**只在以下情况才提示切换：**

| 场景 | 触发条件 | 提示内容 |
|------|---------|---------|
| 图片分析 | 用户贴了截图/图片 | "这个截图分析更适合 MiniMax，要切换到 MiniMax 吗？" |
| 复杂 PRD | 需要从零写一份完整 PRD | "复杂 PRD 更适合 DeepSeek，要切换到 DeepSeek 吗？" |
| 大量 Prompt 工程 | 需要设计多个 Prompt/Skill | "Prompt 设计更适合 DeepSeek，要切换吗？" |

**以下情况不提示：**

| 场景 | 为什么不提示 |
|------|-------------|
| 日常对话/问答 | 所有模型都能做 |
| 简单 PRD/文档 | Claude 也能写 |
| 代码相关 | Claude 最擅长，不切换 |
| Workspace 维护 | 文件操作只有 Claude 能做 |
| 复合任务 | 先做完，不要打断流程 |
| 用户说"你帮我做" | 尊重用户选择 |

---

## 切换建议频率

- 正常情况下，一个对话 session 中建议切换 **0-1 次**
- 只有任务确实明显跨模型能力边界时才提
- 提一次就够了，用户决定切换或不切换，不反复追问
